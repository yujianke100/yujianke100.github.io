用一个例子贯穿全文

整个介绍都围绕同一个真实研究问题展开

贯穿全文的研究问题

"新能源汽车补贴政策，对整个产业链的影响到底有多大？"

要回答这个问题，我们需要四大类数据：

🔵 政策文本 — 全国补贴政策、各省实施细则
🟢 企业数据 — 整车厂财报、电池供应商产能、充电桩布局
🟡 供应链关系 — 谁是比亚迪的电池供应商？供应商的供应商是谁？
🔴 案例数据 — 历史上补贴退坡后的市场反应

❌ 传统做法

一个一个找 → CSV 存本地 → PDF 丢文件夹 → 写 Python 脚本转格式 → 手动查 S3 路径 → 跑 DuckDB……一周过去了，还没开始分析。

✅ GLake 做法

所有数据统一导入 → Catalog 自动管理 → WebUI 浏览 → AI 辅助查询 → 在上面跑分析模型。一下午搞定。

①

存储层放数据

结构化数据（企业财报）→ Parquet 表
非结构化（政策 PDF）→ 原生文件

②

Catalog 管元数据

PostgreSQL 记录路径、标签、权限、README。不存数据本身，只管"数据在哪、是什么"。

③

控制面让你和 AI 操作

WebUI 浏览、API 编程、AI 按规范自动导入数据。

④

计算层做脏活累活

Docker 沙箱做格式转换、数据预处理，不占管理节点资源。

⚡ 关键设计：存储计算解耦

同一个 Catalog，底层存哪里由环境变量决定——

# 开发环境
Catalog → "逻辑路径: /新能源/政策" → 物理存: s3://minio:9000/warehouse/...

# 生产环境
Catalog → "逻辑路径: /新能源/政策" → 物理存: s3://aliyun-oss/warehouse/...

切换只需改一个环境变量，所有代码路径不用动。

传统做法有多痛苦

还是用"新能源汽车政策"的例子来说——

学生：老师，新能源汽车补贴政策的数据在哪？
你：好像在 /data/policy/NEV/ 下……等等我 S3 ls 一下
学生：里面有什么字段？
你：……我不记得了，你跑一下 DuckDB 看看

打开 WebUI，像浏览文件夹一样看到所有数据集。
每个节点有 新能源 · 补贴 · 产业政策 标签和 README。
不用跑任何命令。

拿到新的 CSV → 写 pandas 转 Parquet → boto3 上传 S3 → 调 API 注册 → 手动写 README → 两个月后更新，脚本找不到了 → 重写一遍

告诉 AI 数据在哪 → AI 按模板填表单 → 确认 → AI 自动生成脚本并执行。
人只管 YAML 表单，不碰代码。

政策 PDF、调研报告、会议纪要……散落在微信、邮箱、U 盘、NAS 里。
想搜一份 2023 年浙江的新能源汽车补贴细则？找到算你赢。

PDF/图片和 Parquet 表在 同一个 Catalog 管理，tags + README 全覆盖，WebUI 全文可搜。

要给合作院校老师开放数据？只能给 SSH 账号 → 整个服务器都能看。
或者导 CSV 发邮件 → 版本失控。

5 级 API Key（Admin/Global R/W/Project R/W/Project R），每个 Key 独立 S3 凭证，路径前缀强制隔离。

本地 MinIO 开发好了→上线切阿里云 OSS？所有 s3://minio:9000 要改一遍。
数据预处理吃 CPU → 笔记本卡半天。

逻辑路径+物理路径解耦，切 OSS 只改环境变量。
预处理在独立 Docker 沙箱跑，与管理节点解耦。

你：
把 2023 年新能源汽车补贴政策数据导进来，在 /data/policy/NEV/
AI：
好的，已填写配置模板：
  - 数据源: /data/policy/NEV/ 下的 CSV 和 PDF
  - 格式: CSV → Parquet 转换；PDF 原样存储
  - 目标路径: /新能源政策分析/补贴政策
  - 标签: 新能源补贴产业政策2023
  - Schema 推断: 政策编号(text), 补贴金额(float), ...
请补充或确认以上信息：
你：
标签再加一个"浙江省"，其他确认
⚡ 关键设计：人确认的是 YAML 表单，不是 Python 代码。表单一眼能看懂，改起来几秒钟。
确认表单 → AI 出脚本，这才是高效的人机协作。

→ 数据导入后，WebUI 自动生成可复制的使用脚本 Demo（Python + DuckDB 代码片段）
→ 团队成员通过 API / WebUI / AI 助手就能消费数据

优先方向：政策匹配 & 信息检索

研究"2024 年补贴退坡后，哪些企业受影响最大？"

GLake 的标签系统（新能源·补贴·产业政策 浙江 2023）就是多维索引：

  • Policy-Graph RAG：问"浙江 2023 年补贴标准" → 精准定位到 PDF
  • 政策-案例匹配："某电池厂产能利用率下降" → 自动匹配相关政策做交叉分析

结合 graph 研究经验，一些可以做的方向

以下只是举例说明——GLake 是通用基座，每个方向都能提供数据支撑

🛡️ Agent 市场异构图风险检测

Agent profile 和调用记录存 Parquet 表，描述文本存文档节点。模型通过 API 读 Catalog 数据训练，推理结果写回 tags。

🔗 政策红利的企业网络传导

给定"新能源补贴"种子，沿供应链传播——整车厂受益 → 电池供应商间接受益 → 锂矿厂也受益。

🧩 政策概念异构图对齐

从 main_tag 抽取概念词建图，GNN 聚类消歧。"战略新兴产业 = 新质生产力"——聚类结果写回 Catalog。

📈 时序政策图演化

每月 cron 拉取新政策 → 增量构建时序图 → 做趋势预测和废止预测。

🧠 数据构图本身就是学术方向

从异构数据（表格+PDF+JSON）自动构建知识图谱——这个 数据→图谱 的 pipeline 本身就是研究方向：

  • 异构数据源到统一图谱怎么自动化？
  • 多模态数据如何在图中统一表达？
  • 定时更新时，图谱怎么增量构建？
GLake 提供的不是 AI 模型，而是数据原料 + 规范接口。 上面跑什么模型、做什么分析，老师和团队自己决定。
